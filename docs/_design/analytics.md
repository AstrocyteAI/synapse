# Analytics and insights

This document defines the analytics capabilities in Synapse — what is measured, how it is computed, and where it is surfaced.

---

## 1. Overview

Analytics in Synapse answer questions teams care about:

- Which council members produce the most trusted responses?
- How long does it take to reach a verdict?
- Where is there persistent disagreement?
- Are decisions becoming more or less confident over time?
- What topics are being decided most often?

Analytics are computed from council transcripts stored in Astrocyte and surfaced in the web and desktop UIs. They are read-only; no analytics action modifies council state.

---

## 2. Member performance

Tracks how each named member performs across all councils they participate in.

### 2.1 Metrics per member

| Metric | Definition |
|--------|-----------|
| **Win rate** | % of councils where this member's Stage 1 response ranked #1 in aggregate |
| **Average rank** | Mean aggregate rank position across all participations (lower = better) |
| **Consensus alignment** | % of councils where this member's ranking matched the group consensus |
| **Outlier rate** | % of councils where this member was the dissenting outlier |
| **Council count** | Total participations |

### 2.2 Member leaderboard

Displayed in the web/desktop analytics dashboard. Filterable by:
- Date range
- Council type (llm, agent, mixed)
- Template
- Topic tag

### 2.3 Model comparison

When the same underlying `model_id` is used with different personas or across different councils, performance is tracked at both the model level and the named member level:
- `model_id: openai/gpt-4o` — aggregate across all GPT-4o instances
- `name: The Pragmatist` — specific persona performance

## 3. Decision velocity

Measures how quickly councils move from creation to closed verdict and from verdict to promoted precedent.

### 3.1 Metrics

| Metric | Definition |
|--------|-----------|
| **Council duration** | Time from `council_created` to `session_closed` |
| **Stage durations** | Time per stage (stage_1, stage_2, stage_3) |
| **Time to approval** | Time from `session_closed` to `approved` / `auto_promoted` |
| **Time to precedent** | Time from `session_closed` to `promoted` |
| **Approval rate** | % of councils that receive approval (vs. rejected or timed out) |
| **Auto-promotion rate** | % of councils that bypass approval via confidence threshold |

### 3.2 Velocity trends

Week-over-week and month-over-month trends for:
- Average council duration
- Councils started per day/week
- Verdicts promoted per week

Slow trends signal backlog build-up (pending approvals, unresolved conflicts). Fast trends may indicate insufficient deliberation time.

## 4. Consensus and dissent

### 4.1 Consensus distribution

Histogram of consensus scores across all councils:
- How many councils had high / medium / low consensus?
- Trend over time: is team decision-making becoming more aligned?

### 4.2 Dissent heatmap

Highlights recurring disagreement patterns:
- Which member pairs consistently disagree?
- Which topic tags have the lowest average consensus?
- Which templates produce the most dissent?

Dissent heatmap is most valuable for agent councils where the same named agents participate repeatedly.

### 4.3 Outlier members

Members flagged as outliers in > N% of their participations are highlighted. This may indicate:
- A well-calibrated contrarian perspective (valuable — surface them prominently)
- A misconfigured persona (review their system prompt)
- A model that systematically diverges from others (consider replacing)

## 5. Topic clustering

Automatically groups councils by semantic topic using Astrocyte's entity extraction and embedding capabilities.

**How it works:**
- Council questions and verdict tags are embedded
- Clusters are computed using k-means or HDBSCAN
- Cluster labels are generated by an LLM summarising the cluster's questions

**What it surfaces:**
- Topic clusters and their council counts
- Which clusters have the most pending decisions
- Which clusters have the lowest consensus (contentious topic areas)
- Cluster trend: is decision-making in this topic accelerating or slowing?

**Use case:** an architecture cluster with 12 councils and 0 promoted precedents signals that architectural decisions are being made but not recorded as institutional knowledge.

---

## 6. Confidence trends

Tracks verdict confidence (label and numeric consensus score) over time.

- **Rising confidence** — decisions are becoming more clear-cut; team is aligned
- **Falling confidence** — increasing uncertainty; may indicate a domain undergoing rapid change or a council member configuration that produces more disagreement
- **Per-template confidence** — which templates reliably produce high-confidence verdicts?
- **Per-topic confidence** — which topic areas are most contested?

---

## 7. Knowledge coverage

Measures how well past councils cover the topics teams are currently asking about.

**Coverage score:** for each new council question, Synapse measures how many relevant precedents were available at recall time. Low coverage → teams are making decisions without institutional memory.

**Coverage gaps:** topic clusters with many councils but few precedents indicate decisions being made but not promoted to institutional knowledge.

---

## 8. Dashboard

The analytics dashboard is available in the web and desktop UIs.

**Web UI** (`/analytics`):
- Summary cards: councils this week, average consensus, pending approvals, promoted this month
- Member leaderboard (filterable)
- Consensus distribution chart
- Decision velocity timeline
- Topic cluster map
- Conflict detection history

**Desktop app:**
- All web analytics
- Dissent heatmap (richer, interactive)
- Per-member reasoning trace comparison across councils
- Raw data export (JSON, CSV)

**Messaging integrations:**
- `/analytics` slash command: weekly summary card (council count, average consensus, pending approvals)
- Scheduled weekly digest via notification (see `notifications.md`)

---

## 9. Data source

Analytics are computed from:
- Council session records (status, timestamps, member list, template)
- Stage results stored in Astrocyte `councils` bank
- Verdict metadata (consensus_score, dissent_detected, revision_count)
- Approval and promotion audit trail

Analytics queries use Astrocyte's `recall()` and `reflect()` for semantic grouping and summarisation. Raw metrics (counts, timestamps, durations) are stored in the Synapse operational database (PostgreSQL) for efficient aggregation — not in Astrocyte.

## 10. API

```http
GET /v1/analytics/summary
  ?from=2025-10-01&to=2025-11-01

GET /v1/analytics/members
  ?template=architecture-review&from=2025-10-01

GET /v1/analytics/consensus
  ?tag=architecture

GET /v1/analytics/velocity
  ?granularity=week

GET /v1/analytics/topics
  — topic clusters with council counts and consensus scores

GET /v1/analytics/coverage
  — knowledge coverage gaps
```
